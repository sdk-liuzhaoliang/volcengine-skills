# Provider Versions and Upgrade SOP

This skill uses two Terraform providers during migration:

- New verified examples: `volcengine/volcenginecc ~> 0.0.46`
- Legacy reusable modules: `volcengine/volcengine ~> 0.0.196`

Patch updates within the `0.0.x` line still require a validation plan against at least one applied state because both providers expose fast-moving resource schemas.

## Version pin location

Every module or example declares its own `required_providers` block.

For verified `volcenginecc` examples:

```hcl
terraform {
  required_version = ">= 1.0.7"
  required_providers {
    volcenginecc = {
      source  = "volcengine/volcenginecc"
      version = "~> 0.0.46"
    }
  }
}
```

For legacy modules:

```hcl
terraform {
  required_version = ">= 1.5"
  required_providers {
    volcengine = {
      source  = "volcengine/volcengine"
      version = "~> 0.0.196"
    }
  }
}
```

The skill does not generate a separate `versions.tf` at the root level; consumers inherit the pin via the module they call.

## Upgrade SOP

When a provider release is announced:

1. Fetch the changelog from the provider's GitHub Releases page.
2. Look for breaking changes in resources we use. For `volcenginecc-network`, check `volcenginecc_vpc_vpc`, `volcenginecc_vpc_subnet`, `volcenginecc_vpc_route_table`, `volcenginecc_vpc_security_group`, `volcenginecc_vpc_eip`, `volcenginecc_natgateway_ngw`, `volcenginecc_natgateway_snatentry`, and `volcenginecc_natgateway_dnatentry`; for `volcenginecc-private-nat`, also check `volcenginecc_natgateway_nat_ip` and private NAT `network_type = "intranet"`. For `volcenginecc-vpc-extras`, check `volcenginecc_vpc_prefix_list`, `volcenginecc_vpc_network_acl`, `volcenginecc_vpc_eni`, `volcenginecc_vpc_ha_vip`, and `volcenginecc_vpc_bandwidth_package`; for `volcenginecc-vpc-traffic-mirror-filter`, check `volcenginecc_vpc_traffic_mirror_filter` and `volcenginecc_vpc_traffic_mirror_filter_rule`; IPv6, flow log, mirror target, and mirror session resources are not fully verified yet. For `volcenginecc-ecs`, check `volcenginecc_ecs_keypair`, `volcenginecc_storageebs_volume`, `volcenginecc_ecs_command`, `volcenginecc_ecs_launch_template`, `volcenginecc_ecs_instance`, and `volcenginecc_ecs_invocation`; for ECS extras, check `volcenginecc_ecs_deployment_set`, `volcenginecc_ecs_hpc_cluster`, and `volcenginecc_ecs_launch_template_version`. For EBS examples, check `volcenginecc_storageebs_volume`, `volcenginecc_storageebs_snapshot`, and `volcenginecc_storageebs_snapshot_group`; snapshot groups require attached ECS volumes. For Auto Scaling, check `volcenginecc_autoscaling_scaling_group`, `volcenginecc_autoscaling_scaling_configuration`, `volcenginecc_autoscaling_scaling_lifecycle_hook`, and `volcenginecc_autoscaling_scaling_policy`; the policy is not fully verified. For `volcenginecc-vke`, check `volcenginecc_vke_cluster`, `volcenginecc_vke_node_pool`, `volcenginecc_vke_default_node_pool`, `volcenginecc_vke_addon`, and `volcenginecc_vke_kubeconfig`; `vke_node` is not fully verified yet. For `volcenginecc-tos`, check `volcenginecc_tos_bucket`, `volcenginecc_tos_bucket_cors`, and `volcenginecc_tos_bucket_encryption`. For APIG, check `volcenginecc_apig_gateway` and `volcenginecc_apig_gateway_service`; upstreams and custom domains are not fully verified yet. For VPN, check `volcenginecc_vpn_vpn_gateway`, `volcenginecc_vpn_customer_gateway`, `volcenginecc_vpn_vpn_connection`, and `volcenginecc_vpn_vpn_gateway_route`; SSL VPN client certificates are excluded from shared examples because they store client key material in state. 3. Update the version constraint in the relevant `assets/examples/*/main.tf` or `assets/modules/*/main.tf` files.
4. Run `terraform init -upgrade` in a throwaway test directory.
5. Run `terraform plan` against an existing applied state to verify no spurious diffs.
6. Update this document with the new version and changelog summary.
7. Bump the skill's `version` field in `SKILL.md` (semver patch for compatible upgrades, minor for new modules, major for breaking changes).

## Known compatibility notes

- The provider derives credentials from `VOLCENGINE_ACCESS_KEY` / `VOLCENGINE_SECRET_KEY` / `VOLCENGINE_REGION` natively. The `provider "volcengine" {}` block can be left empty in the root module.
- The `volcenginecc` provider derives credentials from the same environment variables. Use `provider "volcenginecc" {}` with no inline credentials.
- `volcenginecc` generated examples are not always runnable as-is. Check each field against the schema and exclude read-only attributes from resource blocks.
- `volcenginecc_ecs_instance` can show persistent pseudo-diffs on nested Optional+Computed fields even with `ignore_changes = all`; inspect no-op plans manually before applying.
- `volcenginecc_ecs_invocation` should be treated as a second-stage, one-off resource. Wait for Cloud Assistant to register, and be ready to remove completed invocations from state before destroy.
- `volcenginecc_natgateway_dnatentry.external_ip` expects the NAT EIP public address, not the allocation ID. NAT state has short consistency windows after EIP/SNAT/DNAT changes; retry create/destroy when Cloud Control returns `InvalidNatGateway.InvalidStatus`.
- `volcenginecc_vpc_security_group` uses `ingress_permissions` and `egress_permissions`, not `security_group_rules`. New groups can read back a default egress rule once; a convergence apply removed it in the verified `volcenginecc-vpc-extras` example. Standalone `volcenginecc_vpc_eni`, unbound `volcenginecc_vpc_ha_vip`, and empty IPv4 `volcenginecc_vpc_bandwidth_package` are verified. Traffic mirror filters and filter rules are verified standalone. A CLB mirror target can create/delete, but mirror sessions still need an attached ECS ENI from an instance family that supports traffic mirroring; `ecs.g4i.large` failed with `InvalidInstanceSpecification.Malformed`.
- `volcenginecc_natgateway_nat_ip` is verified only with a private NAT gateway (`network_type = "intranet"`, `billing_type = 3`). Public NAT gateways use EIP/SNAT/DNAT instead.
- `volcenginecc_cr_registry` create/delete can each take about two minutes. Keep `status` read-only even though generated docs list it as optional.
- `volcenginecc_vke_cluster` creation is long-running; a no-node cluster took about 9 minutes. `volcenginecc_vke_node_pool` requires real `node_config.security.login` even with zero desired replicas.
- `volcenginecc_vke_addon` with managed `pod-identity-webhook` is verified, but addon create/delete each took about 5.5 minutes. `volcenginecc_vke_default_node_pool` is verified on a zero-worker cluster with only security login and cluster security groups. `volcenginecc_vke_node` still requires an existing ECS instance.
- `volcenginecc_tos_bucket` deletion can remove the bucket while the Cloud Control waiter continues running. Confirm with `tosutil stat` and Terraform refresh before any manual state cleanup.
- `volcenginecc_tls_rule` is verified as a minimal host file rule without `host_group_infos`; set `paths` explicitly and omit `pause` to avoid readback drift. `volcenginecc_tls_consumer_group` is verified with `allow_consume = true` on the topic. `volcenginecc_tls_schedule_sql_task` is verified with indexed source/destination topics and `status = 0`. `volcenginecc_tls_import_task` is verified for TOS source only when `status` is omitted; `status = 4` reads back as `0`. Do not use generated placeholder host group IDs. `volcenginecc_tls_shipper` to TOS can create/delete but is drift-blocked because disabled `status = false` reads back as `true`. `volcenginecc_tls_alarm_notify_group` empty groups are rejected, and a GeneralWebhook receiver group hit provider `receivers` set inconsistency after apply. `volcenginecc_tls_alarm` remains blocked until notification groups are stable. `volcenginecc_cloudmonitor_rule` is verified with a disabled ECS CPU rule, Webhook notification, and `period = "60"`.
- `volcenginecc_redis_parameter_group` is the real resource type; generated docs/examples may show `volcenginecc_redis_parametergroup`, which Terraform rejects.
- Redis passwords remain in Terraform state even when variables are sensitive; never commit Redis example state files.
- `volcenginecc_redis_endpoint_public_address` is verified with a dedicated `volcenginecc_vpc_eip`, but should be used only for deliberate public exposure. Keep Redis allowlists narrow, and destroy the public endpoint before releasing its EIP.
- `volcenginecc_iam_role` trust policies and `volcenginecc_iam_policy` documents should be built with `jsonencode`; pass account-specific IDs through variables instead of hardcoding them in examples.
- `volcenginecc_iam_user` and `volcenginecc_iam_group` are verified without login passwords or access keys. Avoid `volcenginecc_iam_accesskey` in reusable examples because it writes `secret_access_key` into Terraform state.
- `volcenginecc_iam_oidc_provider` is verified with a public HTTPS issuer and lowercase SHA256 thumbprint without colons. Provider creation does not grant access by itself; pair it with a scoped role trust policy for real federation.
- `volcenginecc_iam_saml_provider` is verified for role SSO with base64-encoded SAML metadata containing a public signing certificate. Do not commit SAML private keys; generate or fetch metadata outside the example and pass it as a variable.
- `volcenginecc_ecs_deployment_set` with `strategy = "Availability"` should omit `group_count`; setting `group_count = 7` reads back as `0` and forces replacement. `volcenginecc_ecs_hpc_cluster` is verified as a standalone placement object. `volcenginecc_ecs_launch_template_version` is verified as version 2 of a real launch template; omit empty `ingress_permissions = []` on its security group to avoid endless drift.
